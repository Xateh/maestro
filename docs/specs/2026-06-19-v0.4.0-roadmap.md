## 0.4.0 Theme: Parallel Pipelines & Ecosystem Reach

The reliability platform is complete and hardened. The next frontier is **speed** (run independent verification stages concurrently) and **reach** (more trackers, webhook intake, outbound notifications). Four focused sub-projects:

```
SP7  Concurrent pipeline execution   ← engine architecture
SP8  Real coverage collection        ← fill the SP3 gap
SP9  Tracker expansion & webhooks    ← ecosystem
SP10 Open-loop closure               ← deferred carry-forwards
SP11 Publicity & developer outreach  ← go-to-market
SP12 Ephemeral (agent-authored) workflows ← capstone on SP7 + MCP
```

SP10 is prerequisite-free and can ship first. SP7 is the highest-impact, highest-effort
item and anchors the release. SP12 is the capstone — it consumes SP7's fan-out and the
existing MCP surface, so it lands last. SP11 runs in parallel as a non-engineering
track — it depends only on a tagged release existing, not on the code sub-projects.

## SP7 — Concurrent Pipeline Execution

**Depends on:** item D verdict (done), SP2 (done)

### Problem

The current engine executes one node at a time. In the `full-audit-sweep` pipeline the
four independent verifier stages — `static_analysis`, `review`, `threat_model`,
`edge_cases` — run serially even though they share no data dependencies. A 4-stage
pipeline that takes 4 × 3 min = 12 min wall-clock could be 3 min with fan-out.

The determinism probe (0.3.0 item D) confirmed: **DAG traversal/wiring is
deterministic; LLM outputs are not and never will be.** The North-Star wording is
*auditable / replayable*, not bit-identical. Concurrent execution is compatible with
that contract.

### Design

**Fan-out/join in the LangGraph graph:**

A workflow may declare a `parallel` group — a set of roles with no declared
data dependency between them (no inbound edges from each other). The engine compiles
parallel groups as concurrent LangGraph nodes, collecting their handoffs into a
join node before the next sequential stage.

```yaml
# workflow.json parallel group declaration (new key)
parallel_groups:
  - [static_analysis, review, threat_model, edge_cases]
```

Alternatively, parallelism is inferred automatically when:
- No role in the set has an outgoing edge to another role in the set.
- All roles in the set share the same inbound trigger event.

**Engine changes (`src/langgraph/graph.mjs`, `src/langgraph/engine.mjs`):**

- `buildGraph` detects or reads parallel groups; emits a fan-out node + per-member
  concurrent execution + a `joinNode` that merges the state channels.
- `joinNode` waits for all members to complete (success or terminal failure) before
  emitting the join event. A single member failure does not abort siblings — all
  run to completion and the join records which failed (`parallel_failed[]`).
- State merge: `priorHandoffs` is a map keyed by role; concurrent writes to
  different keys are safe. The join merges all maps.
- `run-manifest.json` records the resolved parallel groups so a `maestro rerun`
  replays the same concurrent structure.

**Concurrency model:** Node.js async — each concurrent role is a `Promise`
(`engine.runRole()`), collected in `Promise.allSettled`. No worker threads, no
child-process fan-out beyond what already exists per role.

**Constraints:**
- A role in a parallel group must not read the handoff of a sibling (only prior
  sequential handoffs). `validateWorkflow` enforces this: an inbound edge from a
  sibling to a group member is a `bad_parallel_group` error.
- `loop_limits` still apply per role; a looping role in a parallel group that
  exhausts its limit emits its fail event before the join — the join sees it as
  a terminal member result.
- `kind: "scoring"` roles cannot be in a parallel group (they read all prior
  handoffs; placing them in a group is a `bad_parallel_group` validation error).

**Observability:**
- Each parallel stage gets its own `stage_event` as today.
- The join emits a synthetic `parallel_join` stage event summarising the group:
  `{group: [role…], duration_ms: max(members), status: "passed"|"partial_failure"}`.
- OTel: the join span links all member spans as parents.

**Default workflow:** unchanged — `default` workflow has no parallel groups.

**`full-audit-sweep` update:** declare `[static_analysis, review, threat_model,
edge_cases]` as a parallel group. `tests` remains sequential (depends on
`edge_cases` handoff). End-to-end pipeline time: ~4× faster for the discovery tier.

### Validation (new rules in `workflow-validate.mjs`)

| Code | Condition |
|------|-----------|
| `bad_parallel_group` | A member has an inbound edge from a sibling in the same group |
| `bad_parallel_group` | A `kind: "scoring"` role is in a parallel group |
| `bad_parallel_group` | A group contains fewer than 2 members |

### Testing

- **Unit — graph compilation:** parallel group → fan-out + join nodes in the compiled
  graph; sequential fallback when no groups declared.
- **Unit — join merge:** concurrent handoff maps merged correctly; partial failure
  recorded; scoring not affected by member order.
- **Integration:** concurrent roles start and finish independently; OTel spans linked.
- **Regression:** all 720+ existing tests pass; single-node workflows unchanged.

## SP8 — Real Coverage Collection

**Depends on:** SP3 (done)

### Problem

`coverage` in the `evaluation` schema is always `{}`. The `min_coverage` gate therefore
**always fails closed** when declared — it finds no evidence and blocks the run.
Coverage data exists on disk (lcov, c8 JSON, pytest `--cov`, Jest coverage reporters)
but SP3 left parsing as future work.

### Design

**Per-command coverage parser declaration:**

```json
{
  "name": "unit tests",
  "run": "npm test -- --coverage --coverageReporters=json",
  "parser": {
    "passed": "Tests:.*?(\\d+) passed",
    "failed": "Tests:.*?(\\d+) failed",
    "total":  "Tests:.*?(\\d+) total",
    "coverage": {
      "format": "c8-json",
      "path": "coverage/coverage-summary.json"
    }
  }
}
```

**Supported `coverage.format` values (first-class parsers):**

| Format | File | Total-lines metric |
|--------|------|--------------------|
| `c8-json` | `coverage/coverage-summary.json` | `total.lines.pct` |
| `lcov` | any `.lcov` / `lcov.info` | `LF`/`LH` ratio |
| `jest-json` | `coverage/coverage-summary.json` | `total.lines.pct` |
| `cobertura` | any `.xml` | `line-rate` attr on `<coverage>` |
| `clover` | any `.xml` | `metrics/@coveredelements / @elements` |
| `regex` | any text file | user-supplied `pct` regex extracting a float |

**Coverage merge:** when multiple commands declare coverage, the `evaluation` handoff
records each command's coverage separately and derives `overall_pct` as the
coverage-weighted average (by total lines when available, else arithmetic mean).

```json
"coverage": {
  "overall_pct": 82.4,
  "by_command": {
    "unit tests": { "pct": 87.2, "format": "c8-json" },
    "integration": { "pct": 71.1, "format": "lcov" }
  }
}
```

**Gate integration:** `scoring.mjs` `correctness_score` sub-score now reads
`coverage.overall_pct` when present and maps to `[0, 1]` (raw percentage / 100).
The existing `min_coverage` gate (`gates.min_coverage`) compares against
`coverage.overall_pct`; a missing or null `overall_pct` still fails closed, as before.

**`commandRunner` changes (`src/command-runner.mjs`):**
- After command exits, reads and parses the declared coverage file (path relative to
  `worktree_path ?? cwd`). Parse errors are recorded as `coverage_parse_error` in
  the command result; the command result itself is unaffected (evidence-only).
- File read is bounded (`assertInsideDir` guard, max 4 MB); large coverage files
  truncate gracefully.

**`bad_command_spec` new validation rule:** `coverage.format` must be one of the
known formats or `"regex"` (which additionally requires a `pct` regex field).

### Testing

- **Unit — parsers:** golden-file snapshots for each format; malformed file →
  graceful `coverage_parse_error`; missing file → `null` coverage.
- **Unit — merge:** weighted average; single-command passthrough; all-null → `{}`.
- **Gate:** `min_coverage: 80` passes at 82.4%; fails at 71%; fails closed on `{}`.
- **Regression:** existing `commands: []` vacuous pass-rate unaffected.

## SP9 — Tracker Expansion & Webhook Intake

**Depends on:** SP0b (done)

### Problem

Maestro's server mode supports exactly one tracker: Linear. GitHub Issues is the
dominant tracker for open-source and solo developers — the primary adoption barrier.
Additionally, external systems (CI, GitHub Actions) can only trigger maestro by polling;
there is no inbound webhook. And there is no outbound notification when a run
completes or halts.

### Design

#### Part A — GitHub Issues tracker

New tracker kind: `github`.

```json
{
  "server": {
    "tracker": {
      "kind": "github",
      "owner": "acme",
      "repo":  "backend",
      "label": "maestro"        // issues with this label are dispatched
    }
  }
}
```

**Polling:** same `server.polling` config (`interval_ms`, `max_concurrent`). GitHub
REST API (`/repos/:owner/:repo/issues?labels=maestro&state=open`). Auth via
`GITHUB_TOKEN` (env or secret store). Rate-limit aware: reads
`x-ratelimit-remaining`; backs off when ≤ 10 remaining.

**Issue mapping → `TaskInput`:** same contract as Linear — `title`, `body`,
`identifier` (`GH-<number>`), `priority` (label `priority:high` etc.), `assignee`.

**Lifecycle write-backs:** on task completion / halt / human-approval-needed, the
tracker adds a comment on the issue with the run summary and, optionally, closes/
labels it (`done_label`, `halt_label` config keys).

**`maestro setup tracker`:** wizard gains a `github` branch — prompts for owner/repo/
label/token, writes config, validates with a test API call.

**`maestro doctor`:** checks `GITHUB_TOKEN` presence and API reachability when
`kind: "github"`.

#### Part B — Inbound webhook intake

New HTTP route on the maestro serve port:

```
POST /api/v1/webhook/:kind
```

Supported `kind` values at launch: `github` (GitHub webhook; validates
`X-Hub-Signature-256`), `generic` (no-auth; bearer token optional via config).

A webhook payload is mapped to a `TaskInput` via a **Liquid template** declared in
`server.webhook_template` (same mechanism as `server.intake_template`). The
dispatched task runs the configured `server.workflow`.

**Security:** signature validation is mandatory for `github`; secret configured in
`server.webhook_secret` (secret store eligible). `generic` webhooks require
`server.webhook_bearer_token` when set, else the route is disabled.

**Config:**

```json
{
  "server": {
    "webhook_kind": "github",
    "webhook_secret": "$MAESTRO_WEBHOOK_SECRET",
    "webhook_template": "{{ payload.pull_request.title }}\n{{ payload.pull_request.body }}"
  }
}
```

#### Part C — Outbound notifications

A run lifecycle hook declared in config:

```json
{
  "server": {
    "notify": {
      "on": ["completed", "halted", "approval_needed"],
      "url": "$MAESTRO_NOTIFY_URL",
      "format": "slack"   // or "generic"
    }
  }
}
```

Supported formats at launch: `slack` (Block Kit message), `generic` (JSON POST with
`{event, task_id, workflow, status, summary}`). Delivery is best-effort (one attempt,
logged to stderr on failure — never breaks a run).

### Testing

- **Unit — GitHub tracker:** mock REST API; label filtering; rate-limit backoff;
  `identifier` mapping.
- **Unit — webhook:** signature validation (valid/invalid/missing); Liquid template
  render; `generic` bearer token check.
- **Unit — notify:** Slack message shape; generic JSON shape; failure is non-fatal.
- **Integration:** existing Linear tracker tests unchanged; `kind: "linear"` default
  preserved.

## SP10 — Open-Loop Closure

**Depends on:** —

Four items, three of them one-file touches.

### SP10a — `require_distinct_reviewer` default-on

In 0.3.0 this shipped as opt-in (`require_distinct_reviewer: false` default). Flip
it: `workflow-validate.mjs` treats `require_distinct_reviewer` as **true when
absent** and emits `non_distinct_reviewer` as a validation **warning** (not error)
for the first release of the default-on behaviour, escalating to error in 0.5.0.
Workflows that set it explicitly to `false` suppress the warning.

`validateWorkflow` changelog: `require_distinct_reviewer` absence now implies `true`;
add deprecation note to any workflow that explicitly sets it `false`.

### SP10b — Real mid-run cancellation

`maestro cancel <id>` today is bookkeeping-only (clears the orchestrator's maps; the
in-flight run continues). Implement real engine-level cancellation:

- `engine.mjs` holds an `AbortController` per active run; stored in a
  `runControllers` map keyed by task id.
- `cancel` command sends `SIGTERM` to the active role's child process and resolves the
  run as `cancelled` (new terminal status). If the child does not exit within the
  existing SIGTERM→SIGKILL grace (2 s), SIGKILL fires.
- Parallel groups (SP7): cancel broadcasts to all live member processes.
- A cancelled run writes its partial handoffs; `maestro artifacts` can inspect them.
- `run-manifest.json` records `cancelled_at`; `maestro rerun` can replay from the
  beginning (not from the cancel point — replay is always a clean re-run).

### SP10c — Per-edge context contract stable

Remove the `experimental_per_edge_context` flag. The feature graduated in 0.3.0
with `KEEP` verdict. New name: `per_edge_context` (without `experimental_`). A
migration shim in the workflow loader accepts the old key with a
`deprecated_experimental_flag` warning for one release.

Stability contract: the `context-contract.mjs` module's public API (`applyContextContract`)
is now stable; no breaking changes without a major version bump.

### SP10d — `coverage` schema non-empty in SP8

SP8 delivers this. Listed here for tracking — once SP8 ships, `coverage: {}`
transitions from a known limitation to documented behaviour (vacuous pass when no
coverage parser configured).

## SP11 — Publicity & Developer Outreach

**Depends on:** a tagged release (0.3.0 ✅); no code dependency.

### Problem

Maestro is a mature reliability platform (SP0–SP6 shipped, 720+ tests) with **no
public-facing presence beyond the repo**. Discovery is limited to people who already
find the GitHub repo. There is no landing page, no hosted docs, no demo a prospective
user can watch in 60 seconds, and no channel for release announcements. The engineering
maturity (0.3.0) is not matched by go-to-market surface — the adoption funnel has no top.

This is a non-engineering track that ships alongside the code sub-projects. It does not
block and is not blocked by SP7–SP10.

### Design

#### Part A — Marketing website / landing page

A static site (single page to start) at a project domain. Sections:

- **Hero:** one-line value prop ("auditable, replayable multi-agent code review
  pipelines") + a primary CTA (install command / "Get started").
- **Problem → solution:** the reliability story (deterministic wiring, gates,
  reviewer independence) in plain language, not spec language.
- **How it works:** a diagram of the pipeline (roles → gates → handoffs) and a short
  annotated `workflow.json`.
- **Social proof / status:** version badge, test count, license, "what's shipped".
- **Footer:** links to docs, GitHub, changelog, security policy.

**Stack:** static-site generator (Astro / Eleventy / plain HTML) hosted on GitHub
Pages or equivalent — no server, no build secrets. Source lives in `docs/site/` or a
dedicated `gh-pages` branch. The site is content-only; no analytics that require
consent banners at launch (keep it simple).

#### Part B — Hosted documentation

Promote the existing `docs/` tree to a browsable, searchable docs site:

- **Getting started:** install, first workflow, first run (the 5-minute path).
- **Concepts:** roles, workflows, gates, handoffs, trackers, server mode.
- **Reference:** `workflow.json` schema, CLI commands (`maestro setup`/`doctor`/
  `serve`/`rerun`/`artifacts`/`cancel`), config keys.
- **Guides:** GitHub tracker setup (SP9), coverage parsers (SP8), parallel groups (SP7)
  — docs land as those features ship.

Generated from the existing Markdown so there is a single source of truth (no
copy-drift between repo docs and the site). Same host as Part A.

#### Part C — README & quickstart polish

The repo README is the most-visited page and the first impression:

- Above-the-fold: badges (version, tests, license, CI), one-line pitch, animated
  demo (asciinema / GIF) of a real run.
- Copy-paste quickstart that works end-to-end in under five minutes.
- Clear links out to the website and hosted docs.

#### Part D — Demo assets

- A **terminal screencast** (asciinema cast committed to the repo + embedded GIF) of a
  full `full-audit-sweep` run, showing gates and the audit trail.
- An **example gallery** under `examples/` — runnable `workflow.json` files
  (default, full-audit-sweep, github-tracker) each with a short README.

#### Part E — Announcement & release channels

- **Changelog as a feed:** `CHANGELOG.md` already exists; publish each tagged release
  as a GitHub Release with curated notes (not just the raw changelog dump).
- **Launch posts:** a short write-up per minor release for the relevant developer
  communities (the where/when is a content decision, not scoped here — this item just
  establishes the cadence and a reusable template).
- **Social/handle presence:** reserve a project handle for release announcements.
  Low-effort, reservation-only at launch.

### Out of scope for 0.4.0

- Paid acquisition, SEO campaigns, sponsorships.
- Consent-gated analytics / tracking pixels.
- A blog engine (a single "announcements" page or GitHub Releases suffices until
  there is volume).

### Done criteria

- Landing page live at a public URL with working install CTA.
- Hosted docs reachable from the landing page, generated from `docs/`.
- README has a demo GIF and a verified ≤5-minute quickstart.
- 0.4.0 published as a curated GitHub Release with notes.

## SP12 — Ephemeral (Agent-Authored) Workflows

**Depends on:** SP7 (fan-out structure), MCP server (`src/mcp/server.mjs`, done),
multi-provider adapters (Codex / Claude / Copilot, done)

### Problem

Today a workflow is authored ahead of time: a human writes `workflow.json` and runs it
by name. But the user already drives maestro from their own agent CLI — via the MCP
server (eight tools, incl. `maestro_create_task`) or a skill/plugin. The natural next
ask is inline, one-off authoring:

> "run a dedicated maestro workflow on this task: `<task>`"

The user's agent should be able to *generate* a workflow tailored to that task, run it
once, monitor it, and discard it — without a committed `workflow.json`. This turns
maestro from a *configured* pipeline into a *programmable* one an agent authors against.

With no `workflow.json` ever written to disk, the user's agent plays three roles in
sequence: **Planner** (free-text task → concrete workflow: roles, SP7 fan-out/join,
gates, per-edge context), **Dispatch agent** (submit the generated workflow object +
task as a run), **Orchestrator** (subscribe to stage events, relay status, drive
lifecycle). The user's own agent *is* the orchestration loop.

### Why this is a release train, not one sub-project

SP12-core (inline submission + run) is small. What makes it *safe and usable* is the
harness around it — and most of that harness is independently valuable, independently
shippable, and not all gated on SP7. Agent-authored workflows declare arbitrary roles,
providers, gates, and **shell commands**: shipping the run path before the guardrails
exist would be RCE-by-design and unbounded-spend-by-design. So SP12 is decomposed into
seven sub-parts spread across the **0.4.x point-release train** *after* the 0.4.0 anchor
(SP7–SP11) ships. Prereqs (authoring, providers, safety, budget) land first and stand
on their own; the run core, orchestration surface, and client reference land last.

| Sub-part | Title | Target | Depends on | Risk |
|----------|-------|--------|------------|------|
| SP12a | Workflow authoring surface | 0.4.1 | MCP ✅, `workflow-validate` ✅ | Low |
| SP12d | Provider registry & preflight | 0.4.1 | adapters ✅, MCP ✅ | Low |
| SP12b | Ephemeral safety policy | 0.4.2 | `command-runner` ✅, workspace sandbox ✅ | Medium |
| SP12c | Budget & resource governance | 0.4.2 | engine, SP7 (concurrency) | Medium |
| SP12e | Ephemeral run core | 0.4.3 | SP7, SP12a–d | High |
| SP12f | Orchestration surface | 0.4.4 | SP12e, SP6a ✅, SP10b | Medium |
| SP12g | Client skill / plugin reference | 0.4.5 | SP12e, SP12f | Low |

**Within-train ship order:** (SP12a ‖ SP12d) → (SP12b ‖ SP12c) → SP12e → SP12f → SP12g.
SP12a/SP12d and SP12b/SP12c are pairwise independent and can each ship in one point
release. SP12e is the convergence point; nothing downstream of it starts until it lands.

---

### SP12a — Workflow authoring surface  *(0.4.1)*

**Depends on:** MCP server ✅, `workflow-validate.mjs` ✅. **No SP7 dependency** — ships
first, useful on its own (lets *any* agent author named workflows too).

So the calling agent can generate a valid workflow cheaply and iterate without burning
runs:

- **Published JSON Schema** for `workflow.json` (`schema/workflow.schema.json`),
  generated from / kept in lockstep with `workflow-validate.mjs` so there is no
  copy-drift. Exposed via an MCP resource so an agent can fetch the contract it must
  author against.
- **`maestro_validate_workflow` MCP tool** — validate-only, no execution. Returns the
  full `validateWorkflow` verdict (error codes + messages). The tight repair loop is
  *generate → validate → fix → revalidate*, all before any role runs.
- **Repair-loop guard:** a server-side `max_validate_attempts` (config, default 5)
  recorded per session so a malformed generator can't spin indefinitely against the
  validate endpoint.
- **Exemplar library** (`examples/` — overlaps SP11 Part D): canonical workflow
  fragments (default, full-audit-sweep, parallel-group, github-tracker) the agent
  composes from rather than authoring freehand.

**Validation:** `bad_workflow_schema` (schema-level rejection, distinct from semantic
`validateWorkflow` codes). **Testing:** schema matches `validateWorkflow` on a golden
corpus (no rule drift); validate-only never mutates state; attempt-guard trips at limit.

---

### SP12d — Provider registry & preflight  *(0.4.1)*

**Depends on:** multi-provider adapters ✅, MCP ✅, `maestro doctor` ✅. **No SP7
dependency.**

So the agent knows what providers exist before planning a fan-out around them:

- **`maestro_list_providers` MCP tool** — enumerates configured adapters (codex,
  claude, copilot, …), each with capability flags (can-plan / can-execute / can-review),
  default permission mode, and **auth status**.
- **Provider preflight** — a doctor-style reachability/auth probe per provider; a
  workflow that names an unconfigured or unauthed provider fails *at plan time* with an
  actionable error, never mid-run.
- **Credential scoping** — per-provider secret resolution through the existing secret
  store; a transient provider role resolves creds at run start, scoped to that run.

**Validation:** `unknown_provider`, `provider_unauthed` (raised during ephemeral plan
validation in SP12e; the registry/preflight primitives land here). **Testing:** mock
adapters with mixed auth states; preflight surfaces missing creds; capability flags
correct per adapter.

---

### SP12b — Ephemeral safety policy  *(0.4.2)*

**Depends on:** `command-runner.mjs` ✅, workspace sandbox ✅. The security gate — **must
ship before SP12e in any shared/hosted mode.**

Agent-authored workflows declare arbitrary `commands[]` (shell) and arbitrary roles.
A server operator must be able to bound what an ephemeral run may do:

```json
{
  "server": {
    "ephemeral": {
      "enabled": true,
      "command_allowlist": ["npm test", "npm run lint", "pytest"],
      "provider_allowlist": ["claude", "codex"],
      "max_fanout": 4,
      "sandbox": "required",
      "gate_relaxation": "forbid"
    }
  }
}
```

- **Command allowlist** — ephemeral `commands[].run` must match the operator allowlist
  (exact or prefix; regex opt-in). A non-matching command is a hard validation reject,
  not a runtime failure.
- **Provider / fan-out caps** — ephemeral runs may only use allowlisted providers and
  may not exceed `max_fanout` members in any parallel group.
- **Sandbox-required** — ephemeral roles always execute in an isolated worktree
  (`worktree_path`), never the live tree, when `sandbox: "required"`.
- **Gate-relaxation policy** — `forbid` (default) means an ephemeral workflow may not
  declare *weaker* gates than the server's baseline (no disabling `require_distinct_reviewer`,
  no lowering `min_coverage` below a floor).
- **Default-closed:** `server.ephemeral.enabled` defaults `false`. Ephemeral submission
  is rejected entirely until an operator opts in.

**Validation:** `ephemeral_disabled`, `command_not_allowlisted`, `provider_not_allowlisted`,
`fanout_exceeds_cap`, `gate_relaxation_forbidden`. **Testing:** each policy rejection;
default-closed blocks submission; sandbox enforcement; allowlist prefix/regex modes.

---

### SP12c — Budget & resource governance  *(0.4.2)*

**Depends on:** engine, SP7 (concurrency model). Ephemeral + heterogeneous providers +
fan-out = agent-decided, unbounded spend and process count. Bound both:

- **Per-run budget caps** — `budget: { tokens?, usd?, wall_clock_ms? }` on the run
  request (and an operator ceiling in `server.ephemeral`). A run that exceeds any cap is
  cancelled (reuses SP10b cancellation) with a `budget_exceeded` terminal status; partial
  handoffs are written.
- **Live cost accounting** — the engine accumulates per-role token/cost usage and emits
  it on the stage-event stream so the orchestrator (SP12f) surfaces running cost and the
  kill-switch can fire mid-run.
- **Concurrency caps + queue** — a global `max_concurrent_roles` (operator config) with
  backpressure: fan-out members beyond the cap queue rather than oversubscribe. Builds
  directly on SP7's `Promise.allSettled` fan-out (bounded pool instead of unbounded).
- **Per-provider rate-limit handling** — generalize the SP9 GitHub rate-limit backoff
  into a shared provider-call limiter so multi-provider fan-out backs off per provider.

**Validation:** `bad_budget_spec` (non-positive cap), `budget_below_floor` (run cap under
an operator minimum is nonsensical — caught early). **Testing:** cap breach cancels and
writes partials; queue respects concurrency cap under fan-out; rate-limit backoff per
provider; cost accounting reflected in stage events.

### SP12e — Ephemeral run core  *(0.4.3)*

**Depends on:** SP7 (fan-out), SP12a (authoring/validate), SP12b (safety), SP12c
(budget), SP12d (providers). The convergence point — the actual inline-run path.

- **`maestro_run_ephemeral` MCP tool** — accepts a full workflow object inline (not a
  name) plus a `TaskInput`. Runs the **same** `workflow-validate.mjs` path *plus* the
  SP12b policy checks and SP12d provider preflight; any failure returns codes to the
  caller (no role runs).
- **Transient roles** — a role materialised for a single ephemeral run, validated
  against the MRC schema but **not persisted** to the role registry. Defined lifetime
  (the run) and isolation (SP12b sandbox); it never mutates the stable MRC registry.
- **Provenance / audit** (north-star compliance) — the `run-manifest.json` for an
  ephemeral run records the **resolved workflow**, the **generation prompt**, and the
  **author identity** (which MCP client / agent submitted it). An agent-authored run is
  thus as auditable and replayable as a committed one.
- **Persist-on-request** — ephemeral workflow is discarded after the run by default; on
  request, the resolved workflow (already in the manifest) is written out as a named
  `workflow.json`. The manifest is the single source of truth; no separate authoring path.

**Validation:** `bad_ephemeral_workflow` (any `validateWorkflow` code, passed through),
`bad_transient_role` (MRC schema fail), plus the SP12b/SP12d codes enforced here.
**Testing:** valid workflow runs / invalid returns codes without running; transient role
used in a parallel group and not persisted; manifest captures prompt + author; persist
round-trips manifest → named `workflow.json` and re-runs identically; existing eight MCP
tools and named-workflow runs unchanged.

### SP12f — Orchestration surface  *(0.4.4)*

**Depends on:** SP12e, stage events (SP6a ✅), cancellation (SP10b). So the calling agent
can *be* the orchestrator without shell access:

- **`maestro_watch_run` MCP tool** — cursor-based tail of the `stage_event` store for a
  run id (poll with an opaque cursor; no new streaming transport needed over MCP stdio).
  Carries gate verdicts, cost (SP12c), and approval-needed halts.
- **Approval over MCP** — `maestro_respond` (approve / reject / answer) so an
  approval-needed halt surfaces to the calling agent → user and can be resolved without
  the TUI/CLI.
- **Lifecycle control over MCP** — expose cancel (SP10b), and add pause / resume /
  retry-a-role so the orchestrator drives, not just observes.
- **Re-attach across sessions** — `maestro_watch_run` by id works from a fresh session
  (server-mode task persistence already survives a client disconnect); a dropped agent
  reconnects to a live run.

**Validation:** `bad_watch_cursor`, `bad_lifecycle_action` (action illegal for current
run state). **Testing:** cursor tail returns events in order with no gaps/dupes; approval
round-trip resolves a halt; cancel/pause/resume transitions; re-attach from a new client.

### SP12g — Client skill / plugin reference  *(0.4.5)*

**Depends on:** SP12e, SP12f. The capstone — the thing the *user* actually runs.

A reference client (maestro **skill / MCP-client plugin**) that wraps the SP12a–f tools
into the planner → dispatch → orchestrator loop, so users don't reinvent it:

- Triggers on "run a dedicated maestro workflow on this task: …".
- Fetches the schema (SP12a) + provider registry (SP12d), generates a workflow,

- Ships as documented, runnable reference (under `examples/` and the SP11 docs site),

**Testing:** end-to-end — reference client plans → runs → orchestrates a
heterogeneous-provider fan-out to completion against a live server; persist-on-request
produces a re-runnable named workflow.

## Future — Extensible Hook System (0.5.x)

**Status:** Candidate, not scheduled. **Gate:** a real demand signal (same bar as
*Role registry*). Recorded here because the substrate is already in place, so this can
be picked up cleanly whenever demand arrives.

### Why deferred (not in 0.4.0)

Maestro today has one hook surface: four **global, workspace-scoped** shell hooks —
`after_create` / `before_run` / `after_run` / `before_remove` — run by `src/workspace.mjs`
around each per-issue workspace (`server.hooks`). A full per-role / per-stage / matcher
system is **net-new public config surface** (a stability commitment maestro must carry
forward), with no user currently pulling on it. By the roadmap's own standard — *Role
registry* deferred for "needs demand signal" — this waits. 0.4.0 is anchored by speed
(SP7) and reach (SP9); adding a speculative subsystem dilutes that focus.

### Why it's cheap when demanded

The two pieces a hook system needs already exist:
- **Lifecycle exec** — `src/workspace.mjs` already spawns shell hooks with
  timeout/capture (`runHook`); the logic just needs extracting into a shared runner.
- **Event stream** — the engine already emits a `stage_event` for every step
  (`src/stage-events.mjs`, SP6a); per-stage hooks subscribe to it rather than inventing
  new instrumentation.

### Increments (ship smallest-first, each independently shippable)

Split into three so the high-value, low-risk piece (per-role hooks) can land without
committing to the whole taxonomy.

#### HK1 — Hook runner + per-role hooks  *(foundation; lowest risk)*

**Depends on:** SP0b ✅. The single highest-value slice — "run X before/after a specific
role" is the most-requested gap.

- Extract the shell-exec logic from `src/workspace.mjs` (`runHook`: spawn, timeout,
  capture) into a shared `src/hooks.mjs` `runHooks(event, ctx)`; `workspace.mjs` keeps
  its four call sites but delegates execution.
- Add role-scoped hooks in `workflow.json` — a role's `hooks[]` of
  `{ event, run, timeout_ms?, blocking? }` for `before_role` / `after_role` /
  `on_role_fail`.
- Engine wiring (`src/langgraph/engine.mjs`): `runRole` fires the three role events,
  injecting `MAESTRO_*` env (`MAESTRO_TASK_ID`, `MAESTRO_ROLE`, `MAESTRO_EVENT`,
  `MAESTRO_WORKFLOW`, `MAESTRO_WORKSPACE`).
- **Blocking semantics:** `before_role` defaults `blocking: true` (non-zero exit aborts
  the role with a `hook_blocked` terminal status); `after_role` / `on_role_fail` default
  `blocking: false` (logged, recorded, never break a run — matches the SP9 `notify`
  contract).
- Hook firings recorded as `hook_result` entries on the run manifest (audit-only; hooks
  are side-effecting and are **not** replayed on `maestro rerun`).
- **Validation** (`workflow-validate.mjs`): `bad_hook_event`, `bad_hook_spec`
  (missing `run` / non-positive `timeout_ms`).
- **Backward compat:** existing `server.hooks` object shape untouched.

#### HK2 — Stage-event subscription hooks + matcher  *(the claude-code-style surface)*

**Depends on:** HK1 (runner), SP6a ✅ (stage events).

- New event `on_stage_event` with an optional `match` object filtering the `stage_event`
  projection (`{ kind, status }`) — the *event × matcher* matrix that maps onto Claude
  Code's `PreToolUse`/matcher model.
- Convenience events `on_gate_fail` and `on_approval_needed` as sugar over common
  `match` filters.
- The engine's stage-event emit path fires these after each event is persisted; the
  `stage_event` JSON is piped to the hook on stdin.
- **Validation:** `bad_hook_match` (`match` on a non-`on_stage_event` hook).

#### HK3 — Scope ladder (global + workflow scopes)  *(organizing layer; do last)*

**Depends on:** HK1, HK2.

- Promote hooks beyond the role scope: global `server.hooks.events[]` (additive to the
  existing four) and workflow-level `hooks[]` in `workflow.json`.
- Ordering contract: all matching hooks fire, ordered **global → workflow → role**.
- **Validation:** `bad_hook_scope` (an event declared at a scope it doesn't support,
  e.g. `after_create` on a role).

### Relationship to SP9 `notify`

SP9 Part C (`server.notify`) ships standalone in 0.4.0 with its own small dispatch.
HK3's global event scope is the natural future home for that dispatch — when HK lands,
`notify` can be re-expressed as a built-in `on_*` hook with an HTTP target rather than a
parallel mechanism. Not a 0.4.0 dependency in either direction.

## Future — Process Recovery & Durable Resume (0.5.x)

**Status:** Candidate, scheduled-leaning. **Gate:** the 0.4.x ephemeral train (esp. SP12e)
landing, since agent-authored long-running fan-outs are exactly the workloads that make
mid-run crash loss expensive. Supersedes the lone "Concurrent engine mid-run state
persistence" deferral in *Not in 0.4.0* and gives it a real shape.

### Problem

Maestro runs **many kinds of process** — the orchestration engine, spawned provider-CLI
subprocesses, the server-mode daemon, the MCP server, the TUI, and per-issue workspaces.
Today recovery is uneven: queued tasks resume (`resumeQueuedTask`), the SQLite task store and
`run-manifest.json` / stage-events persist, but a process that dies **mid-role** loses that
role's in-flight work and can leave orphaned agent subprocesses or tasks stuck in a
non-terminal state with no owner. There is no single contract for *what each process
guarantees on crash/restart/disconnect*. This sub-project defines that contract and closes the
durability gaps under it.

### Recovery taxonomy (the contract)

Every long-lived or spawned process is assigned exactly one **recovery class**, and the
durable state it relies on is named:

| Class | Meaning | Example processes |
|-------|---------|-------------------|
| **resumable** | On restart, continues from the last durable checkpoint without redoing completed, side-effecting work | orchestration engine / task run |
| **reattach-only** | Holds no authoritative state; reconnects to a live run by id | TUI, MCP client, CLI watcher |
| **restart-clean** | Safe to kill and relaunch; rebuilds state from the store | server-mode daemon, MCP server |
| **supervised-child** | Owned by a parent; on parent death is reaped or adopted, never double-spawned | provider-CLI agent subprocesses |
| **fire-and-forget** | Idempotent side-effect; re-running is a no-op | notify dispatch, audit writes |

### Increments (smallest-first, each independently shippable)

#### RC1 — Recovery taxonomy & inventory  *(foundation; doc + registry)*

**Depends on:** —. Enumerate every process maestro spawns or hosts, assign its recovery class
and the durable state it depends on, and encode that inventory as a small in-repo registry. A
test asserts every spawn/serve entry point is classified (no unclassified process). Pure
contract — no behavior change. This is the spec the rest of RC implements against.

#### RC2 — Durable run checkpoints  *(the 0.5.0 mid-run item)*

**Depends on:** SP7 (engine), RC1. The engine writes a checkpoint at each **role boundary**
(role completed + handoff written) into the run record, so a crashed run resumes from the last
completed role rather than from the start. Builds directly on already-persisted substrate
(`run-manifest.json`, stage-events, SQLite) — no new store. **Idempotency:** completed roles
are not re-run; side-effecting roles (`commands[]`, git actions) are gated by recorded
completion markers so resume never double-executes. `maestro rerun`/resume semantics made
explicit. **Validation/status:** an interrupted run carries a distinct terminal-or-resumable
status (`interrupted`) with partial handoffs intact.

#### RC3 — Agent subprocess supervision & orphan reaping  *(supervised-child class)*

**Depends on:** RC1. Track spawned agent-subprocess PIDs in the run record; on parent restart,
detect orphans and either reap them or adopt their output, and never double-spawn a role
that's already running. Closes the "zombie agent after a crashed engine" gap.

#### RC4 — Startup reconciliation & crash-safe writes  *(restart-clean + durability)*

**Depends on:** RC2, RC3. A startup reconciliation pass scans the task store for tasks in
non-terminal states whose owning process is gone, and either resumes (RC2) or marks them
`interrupted` with partials written. All authoritative writes (manifest, config, secrets,
checkpoints) become atomic (temp-write + rename) so a crash mid-write can't corrupt state.

#### RC5 — Uniform client reattach  *(reattach-only class)*

**Depends on:** RC1, SP12f (`maestro_watch_run`). Generalize SP12f's by-id reattach into one
contract shared by TUI, CLI, and MCP: any client reconnects to a live run by id, server-mode
persistence already surviving the disconnect. Removes per-client bespoke reconnect logic.

### Relationship to the 0.4.x train

RC2 is the durable backbone the ephemeral run core (SP12e) and budget kill-switch (SP12c)
both lean on — a `budget_exceeded` or cancelled run must leave a resumable/`interrupted`
checkpoint, not a black hole. RC5 is the cross-process generalization of SP12f's single-tool
reattach. RC therefore lands **after** the 0.4.x train, consuming its primitives rather than
blocking them.
