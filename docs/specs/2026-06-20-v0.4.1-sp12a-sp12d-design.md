# v0.4.1 — SP12a + SP12d design (authoring surface + provider registry)

**Target:** 0.4.1 point release. **Depends on:** MCP server ✅, `workflow-validate.mjs` ✅,
multi-provider adapters ✅, `maestro doctor` ✅. **No SP7 dependency.** Both sub-parts are
low-risk, independently useful, and ship together as the 0.4.1 train stop (see
`2026-06-19-v0.4.0-roadmap.md`).

Goal: let *any* MCP-client agent author a named workflow and discover providers **cheaply
and offline**, before SP12e ever runs a role. Nothing here executes a workflow — these are
the authoring/discovery primitives the run core (0.4.3) consumes.

---

## SP12a — Workflow authoring surface

### A1. Published JSON Schema — `schema/workflow.schema.json`

A hand-authored JSON Schema (draft 2020-12) describing the **structural** shape of
`workflow.json`: `initial`, `roles{}` (provider, alias, model, fallback[], tools[],
deny_tools[], commands[], max_visits, verifies, output_schema…), `transitions{}`,
`modes{}`, `parallel_groups[]`, `gates{}`, `edge_context{}`, `loop_limits`. Structure only —
it deliberately does **not** encode semantics (reachability, cycle-termination,
distinct-reviewer, cross-provider). Those stay owned by `validateWorkflow`.

**Lockstep without a generator.** `validateWorkflow` is imperative branching; a faithful
schema generator would be brittle. Instead a **drift test** (`maestro-workflow-schema.test.mjs`)
asserts schema-vs-validator **accept/reject parity** on a golden corpus: for every fixture,
`schema.validate(fixture).ok === validateWorkflow(fixture).ok` *at the structural level*
(fixtures that are only semantically invalid are tagged so the structural-parity assertion
skips them). Adding a workflow field requires adding a corpus fixture that exercises it — the
test is the no-copy-drift guarantee. CI red on divergence.

The schema is a published artifact (servable as-is, no build step).

### A2. `maestro_validate_workflow` — inline + disk (extend existing tool)

Extend the existing tool with an optional `workflow` argument:

- **no arg** → read `.maestro/workflow.json` (today's source, unchanged — back-compat).
- **`workflow` object** → validate the inline candidate without touching disk.

**Structural pre-check runs on both paths.** Whichever source the workflow comes from
(disk or inline), the pipeline is identical: (1) **structural** check against
`schema/workflow.schema.json` via the existing `schemas/index.mjs` ajv seam — a structural
failure returns the new code **`bad_workflow_schema`** (distinct from semantic
`validateWorkflow` codes) and short-circuits before the semantic pass; (2) on structural
pass, run the full `validateWorkflow` and return its `{ok, errors, warnings}` verdict. Same
verdict for the same workflow regardless of source. The tight repair loop is *generate →
validate → fix → revalidate*, all before any role runs. Validate-only **never mutates
state**.

*Back-compat note:* disk-mode callers now also get `bad_workflow_schema` for a structurally
malformed `workflow.json` instead of a semantic-only verdict. This is a strictly stronger,
additive rejection (a structurally-bad disk workflow would have failed the semantic pass
anyway, just with less precise codes); the `{ok, errors, warnings}` shape is unchanged. The
golden corpus pins this so the disk path can't silently regress.

### A3. Repair-loop guard — `max_validate_attempts` (with explicit recovery)

Server-side counter so a malformed generator can't spin indefinitely against the validate
endpoint. Config: `server.mcp.max_validate_attempts` (default **5**) and
`server.mcp.validate_cooldown_ms` (default **60000**). Counts **consecutive failed**
inline-validate attempts, **per MCP session**, in server process memory (keyed by session id).

Disk-mode (no `workflow` arg) calls are exempt — they don't drive a repair loop, so they
neither increment nor are blocked by the counter.

**Recovery — three independent ways the guard clears (no deadlock):**

1. **On success** — any inline validate that returns `ok:true` resets the counter to 0. The
   normal exit: the loop converged, the throttle forgets the failures.
2. **On cooldown** — once the counter hits the max, further inline validates are rejected with
   the verdict code **`validate_attempts_exhausted`** *until* `validate_cooldown_ms` has
   elapsed since the last attempt; the next attempt after the cooldown resets the counter and
   validates normally. This is what breaks the would-be deadlock — you cannot "validate your
   way out" while blocked, so time, not another call, clears it. The rejection verdict carries
   `retry_after_ms` so the agent knows exactly how long to back off.
3. **On session/process end** — counter is process-memory; a new MCP session (or server
   restart) starts clean.

The exhausted verdict is a normal `{ok:false, errors:[{code:"validate_attempts_exhausted",
retry_after_ms}], warnings:[]}` shape — it never throws, never mutates state, and is itself
not counted as an attempt.

### A4. Schema as an MCP resource

Expose `schema/workflow.schema.json` via an MCP **resource** so an agent fetches the contract
it authors against. Adds `resources: {}` to server capabilities + `ListResources` /
`ReadResource` handlers. One resource URI, e.g. `maestro://schema/workflow.json`. Read-only.

### A5. Exemplar library — `examples/`

Canonical, validating workflow fragments the agent composes from rather than authoring
freehand (overlaps SP11 Part D docs): `examples/default.workflow.json`,
`examples/full-audit-sweep.workflow.json`, `examples/parallel-group.workflow.json`,
`examples/github-tracker.workflow.json`. Each is asserted valid (`validateWorkflow().ok`) by
the drift/corpus test so exemplars can't rot.

### A6. New validation codes

- `bad_workflow_schema` — structural (schema-level) rejection, distinct from semantic
  `validateWorkflow` codes. Raised on both disk and inline paths.
- `validate_attempts_exhausted` — repair-loop guard tripped; carries `retry_after_ms`.

Both documented in `docs/mcp.md`.

**Testing (SP12a):** schema↔validator parity on the golden corpus (no rule drift);
structural pre-check fires on **both** disk and inline paths and yields the same verdict for
the same workflow; validate-only never mutates state; attempt-guard trips at the limit, and
recovers on success **and** after `validate_cooldown_ms` **and** on a fresh session;
disk-mode is exempt from the counter; every exemplar validates; resource read returns the
schema bytes.

## SP12d — Provider registry & preflight

### D1. Capability flags — `src/adapters/capabilities.mjs`

Per-provider capability flags as an **open map** (`{ plan, execute, review, … }`) so future
multimodal caps (`image_gen`, `audio_gen`) slot in with no schema change. Source of truth is
a **hybrid**: a static `BUILTIN_CAPS` table per built-in adapter (claude/codex/copilot/gemini/
antigravity/ollama — mirroring `ENFORCEMENT_BY_PROVIDER`'s shape) supplies sane defaults, and
a provider def's `capabilities` object in config **overrides/extends** it:

```js
effectiveCaps = { ...BUILTIN_CAPS[adapterBase], ...(providerDef.capabilities ?? {}) }
// custom adapters with no built-in entry default to { plan:true, execute:true, review:true }
```

`resolveCapabilities(providerDef, adapterId)` returns the merged map. No config burden for
built-ins; operators opt in to overrides only when needed.

### D2. `maestro_list_providers` MCP tool

Enumerates configured adapters from `config.providers`, each with:
- `provider` key, resolved `adapter` id, `default_alias`, `models[]`;
- **capability flags** (D1);
- **default permission mode** (provider def's permission / adapter default, validated against
  `VALID_MODES`);
- **auth status** (D3).

Read-only, offline-capable. Built in `src/provider-registry.mjs` (`listProviders({config, cwd,
env})`) so the MCP handler stays thin and the logic is unit-testable without the server.

### D3. Provider preflight (best-effort auth status)

A doctor-style reachability/auth probe per provider, reusing existing primitives — **no new
probe infra, no new secret store**:
- reachability via `commandAvailable` (`provider-availability.mjs`);
- credential presence via `keys.mjs` (`resolveProviderEnv` / `resolveAliasEnv`) against the
  existing secret store — resolvable ⇒ creds present.

Status enum: **`ready`** (CLI resolves + creds resolvable), **`missing_cli`** (command not
found), **`missing_creds`** (CLI present, required env unresolved), **`disabled`**
(`enabled:false`), **`unknown`** (can't determine). Deep token/online auth validation is
explicitly **deferred** — best-effort is enough to fail a fan-out plan at plan time (SP12e)
rather than mid-run.

### D4. Credential scoping (primitive only)

Per-provider secret resolution already flows through the existing secret store
(`keys.mjs`/`loadLocalSecrets`); D3 surfaces *resolvability*. Run-scoped resolution for a
transient role is **consumed** in SP12e — here we only expose that a provider's creds resolve.

### D5. Validation codes (primitives land here, raised in SP12e)

`unknown_provider`, `provider_unauthed` — the registry/preflight data lands in 0.4.1; these
codes are *raised* during ephemeral plan validation in SP12e (0.4.3). 0.4.1 ships the
detection primitives, not the rejection path.

**Testing (SP12d):** mock adapters with mixed auth states — preflight surfaces missing CLI vs
missing creds vs disabled distinctly; capability flags correct per built-in adapter and config
override merges; custom adapter defaults to all-true; `list_providers` shape stable; offline
(no network) returns best-effort statuses without hanging.
